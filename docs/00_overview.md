# Lesson Track - 個別指導 学習管理アプリ 概要

## 1. アプリの目的

個別指導における学習進捗を一元管理し、以下を実現する：

- **講師間の引き継ぎ**：どの生徒がどこまで進んだかを即座に把握
- **保護者への共有**：学習結果をPDF/メールで報告
- **記録の簡便化**：スマホで答案を撮影するだけで点数を自動記録

## 2. ユーザーと規模

| ユーザー | 人数 | 役割 |
|---------|------|------|
| 管理者 | 1名 | 目標設定、生徒・講師管理、保護者対応 |
| 講師 | 複数名 | 授業実施、答案撮影・記録、進捗確認 |
| 保護者 | 生徒数分 | 学習結果の閲覧（PDF/メール受領） |
| 生徒 | 2名（最大5名） | アプリへの直接アクセスなし |

## 3. 教材（コンテンツ）の構成

教材会社の既製プリントを使用。単元名・ステップ番号がプリントに印刷済み。

### 漢字・単語・熟語

| ステップ | 内容 | 運用 |
|---------|------|------|
| ラーニング | 新しい漢字・単語・熟語の一覧 | 宿題として渡す |
| ステップ1 | 読み・日本語訳のテスト | **任意実施**（必要な生徒のみ補助的に使用） |
| ステップ2 | 書き・英単語/熟語を書くテスト | 次回授業時にテスト実施（**完了判定はステップ2基準**） |

> **確定した運用ルール**:
> - 通常フローは「宿題ラーニング → 次回ステップ2テスト」の2ステップ
> - ステップ1は基本スキップ。定着が不十分な生徒にのみ補助的に実施
> - ステップ1を実施した場合のみ記録対象（実施しなければ記録不要）
> - 進捗データモデル: `step1_status = not_required | done | skipped`

### 英語・数学

| ステップ | 内容 | 運用 |
|---------|------|------|
| ラーニング | 説明・解説 | 授業中に使用 |
| ステップ1 | 問題 | 生徒が解く→講師が採点→間違いは再挑戦 |
| ステップ2 | 類題 | ステップ1で間違えた単元のみ実施 |

## 4. 主要機能

### 4.1 認証・ユーザー管理
- 講師ログイン（誰が対応したか記録するため）
- 管理者権限（生徒・講師の登録、目標設定）

### 4.2 生徒管理
- 生徒の基本情報（名前、学年、受講科目）
- 受講パターン（春期講習のみ / 継続受講）

### 4.3 学習目標・計画
- 管理者が「何をいつまでにやるか」を設定
- 進捗に応じて目標を変更可能
- カリキュラムの進捗率を可視化

### 4.4 授業記録（メイン機能）
**記録フロー：**
1. 講師がWebアプリで生徒を選択
2. 実施した単元・ステップを選択（プリント上の情報と対応）
3. スマホで答案を撮影・アップロード
4. Gemini APIで画像解析 → **講師が採点済みの答案から点数欄を読み取る**（自動採点ではない）
5. 講師が点数を確認・修正して保存（AI読み取りは補助。最終判断は講師）
6. コメント（メモ）を任意で追加

### 4.5 進捗確認
- 生徒ごとの学習履歴一覧（いつ・何を・何点）
- 単元ごとの進捗状況（完了基準は要件定義で別途定義）
- 次にやるべきプリントの提示（提示ロジックは要件定義で別途定義）

> **要件定義で決めるべき項目**:
> - 完了判定基準（例：ステップ2で○点以上なら完了）
> - 再挑戦の上限回数
> - 未完了時の次ステップへの遷移ルール

### 4.6 保護者共有
- 学習結果一覧をPDFで出力
- メールで送信

## 5. 技術構成

| 項目 | 技術 |
|------|------|
| フロントエンド | Next.js 16 / React 19 / TypeScript / Tailwind CSS |
| バックエンド | Next.js API Routes (App Router) |
| データベース | Supabase（PostgreSQL + Auth + Storage） |
| 画像解析 | Google Gemini API（実装時点の最新 Flash モデルを採用） |
| PDF生成 | サーバーサイドでPDF生成ライブラリを使用 |
| ホスティング | Vercel |

### なぜ Supabase か
- 認証（講師ログイン）が組み込み
- 画像ストレージが組み込み（答案写真の保存）
- PostgreSQLベースで柔軟なクエリ
- 無料枠が十分（この規模なら無料枠内で運用可能）
- Row Level Security（RLS）で講師ごとのアクセス制御が可能

### 個人情報の取り扱い
未成年の個人データを扱うため、以下を要件定義で明確化する：

| 項目 | 方針（要件定義で詳細化） |
|------|------------------------|
| 保存するデータ | 氏名、学年、受講科目、学習記録、答案画像 |
| データ保存期間 | 退塾後○ヶ月で自動削除（期間は要決定） |
| 閲覧権限 | 管理者：全生徒 / 講師：担当生徒のみ or 全生徒（要決定） |
| 答案画像の扱い | Supabase Storage にアップロード、外部公開しない |
| 削除ルール | 管理者が手動で生徒データを削除可能にする |

## 6. Gemini API 費用見積もり

実装時点の最新 Flash モデルを採用する。モデルの廃止・料金改定に備え、モデル名は環境変数で管理する。

> **注意**: Gemini 2.0 Flash は 2026-06-01 に廃止予定。
> 現時点での推奨候補は **gemini-2.5-flash-lite** など。実装開始時に最新の公式ドキュメントで確認する。

| 項目 | 数値 |
|------|------|
| 月間画像枚数 | 最大50枚 |
| 予算上限 | $8/月 |
| **見込み** | **Flash系モデルの場合、50枚/月は$1未満で収まる見通し** |

> 無料枠のレート制限はプロジェクト・ティアにより異なるため、AI Studio上で実際の値を確認すること。
> 料金の詳細: https://ai.google.dev/gemini-api/docs/pricing
> レート制限: https://ai.google.dev/gemini-api/docs/rate-limits

## 7. 現在の生徒情報

### 生徒A（春期講習のみ）
- 4月から中学1年生
- 春期講習：4日間（1日1時間、計4時間）
- 実施済み：なし

### 生徒B（継続受講）
- 既に6回（3日間6時間）の授業を実施済み → 過去データは手入力で登録
- 春期講習：4日間（1日1時間×4時間）
- 4月以降：週1回（1回2時間）継続

## 8. 画面イメージ（主要画面）

1. **ログイン画面** - 講師認証
2. **ダッシュボード** - 今日の予定、生徒一覧
3. **生徒詳細画面** - 学習履歴、進捗、次のプリント
4. **授業記録画面** - 生徒選択→単元選択→写真撮影→点数確認→保存
5. **目標管理画面**（管理者）- カリキュラム設定、目標変更
6. **PDF出力画面** - 保護者向けレポート生成
