# Lesson Track - 要件定義書

## 1. データモデル

### 1.1 ER図（テーブル一覧）

```
users (講師・管理者)
  │
  ├── students (生徒)
  │     │
  │     ├── lesson_records (授業記録)
  │     │     ├── record_images (答案画像)
  │     │     └── → units (単元)
  │     │
  │     ├── learning_goals (学習目標)
  │     │     └── → units (単元)
  │     │
  │     └── student_subjects (受講科目)
  │           └── → subjects (科目)
  │
  └── subjects (科目マスタ)
        └── content_groups (教材グループ)
              └── units (単元マスタ)
```

> **将来の拡張**: 生徒ごとに使用する教材グループを個別制御する必要が出た場合、
> `student_content_groups` テーブルを追加して対応する。フェーズ1では `student_subjects`（科目単位の紐付け）のみ。
>
> **フェーズ1の教材グループ表示ルール**: 生徒が受講する科目に属する**全教材グループを表示**する。
> 授業記録の有無によるフィルタリングは行わない。少人数運用（2〜5名）のため、表示量が問題になることはなく、
> 講師が次に使う教材を判断する際にも未使用の教材グループが見えた方が便利である。
>
> **教材グループの追加予定**: 「英文法入門」（84項目）を `content_groups` に追加予定。
> 単元データの提供を受け次第、`content_groups` + `units` にシードデータを投入する。
> スキーマ変更は不要（既存の構造で対応可能）。

### 1.2 テーブル詳細

#### users（講師・管理者）
Supabase Auth を使用。追加プロフィール情報を別テーブルで管理。

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid (PK) | Supabase Auth の user id |
| display_name | text | 表示名 |
| role | enum | `admin` / `instructor` |
| avatar_url | text? | プロフィール画像URL（任意） |
| created_at | timestamptz | 作成日時 |

#### students（生徒）

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid (PK) | |
| name | text | 氏名 |
| grade | text | 学年（例: `中1`, `中2`） |
| enrollment_type | enum | `spring_course` / `ongoing` / `trial` |
| schedule_note | text? | 受講パターンの補足メモ（例:「週1回2時間」）。表示用のみ、ロジックには使用しない |
| is_active | boolean | 在籍中かどうか |
| created_at | timestamptz | |
| created_by | uuid (FK → users) | 登録した管理者 |

#### subjects（科目マスタ）

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid (PK) | |
| name | text | 科目名（例: `英語`, `数学`, `国語`, `理科`, `社会`, `算数`） |
| display_order | int | 表示順 |

> **変更点**: `category` カラムを削除。同一科目内に vocabulary 系・academic 系の教材が混在するため（例: 英語の中に英単語と英文法）、category は教材グループ側で管理する。

#### content_groups（教材グループ）

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid (PK) | |
| subject_id | uuid (FK → subjects) | 所属科目 |
| name | text | 教材名（例: `英文法`, `英熟語 高校入試重要300`, `1年のまとめ`） |
| category | enum | `vocabulary`（漢字・単語・熟語系） / `academic`（英文法・数学系） |
| grade | text? | コンテンツの対象学年（例: `中1`, `中2`）。学年横断の場合は null |
| display_order | int | 科目内の表示順 |
| created_at | timestamptz | |

> **grade の意味**: コンテンツの内容レベルを示すラベルであり、生徒の学年とは独立している。中3の生徒が `中1` レベルの教材で復習する、中1の生徒が `中2` の教材に先取りで取り組む、といった運用が前提。生徒の学年による教材グループのフィルタリングは行わない。
> **category の意味**: 進捗ロジック（完了判定・次プリント提示）の分岐に使用。vocabulary 系はラーニング→Step2、academic 系はラーニング→Step1→Step2 のフローとなる。

#### units（単元マスタ）

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid (PK) | |
| content_group_id | uuid (FK → content_groups) | 所属教材グループ |
| name | text | 単元名（プリント上の表記と一致させる） |
| unit_number | int | 単元番号（教材グループ内での並び順） |

#### student_subjects（生徒の受講科目）

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid (PK) | |
| student_id | uuid (FK → students) | |
| subject_id | uuid (FK → subjects) | |

#### lesson_records（授業記録）

| カラム | 型 | 説明 |
|--------|------|------|
| id | uuid (PK) | |
| student_id | uuid (FK → students) | |
| unit_id | uuid (FK → units) | |
| instructor_id | uuid (FK → users) | 担当講師 |
| lesson_date | date | 授業実施日 |
| step_type | enum | `learning` / `step1` / `step2` |
| score | int? | 点数（nullの場合は採点なし＝ラーニング配布等） |
| max_score | int? | 満点（プリントごとに異なる） |
| score_source | enum? | `manual` / `ai_extracted` / `ai_corrected`（score が null の場合は null） |
| completion_type | enum? | `passed`（80%以上）/ `step1_perfect`（S1満点によりS2スキップ）/ null（未完了） |
| comment | text? | 講師コメント |
| created_at | timestamptz | |

> **score_source**: 手動入力かAI読み取りか、AI読み取り後に講師が修正したかを区別。score が null（ラーニング配布等）の場合は score_source も null。
> **completion_type**: 単元の完了根拠を記録。`passed` = ステップ2で80%以上、`step1_perfect` = ステップ1満点によりステップ2スキップ。未完了レコードは null。
> **レコード作成 = 実施済み**: lesson_records の行が存在すること自体が「実施した」という事実。未実施の単元はレコードが存在しない。

### 1.3 制約・インデックス・削除方針

#### 一意制約
| テーブル | カラム | 説明 |
|---------|--------|------|
| content_groups | `(subject_id, display_order)` | 同一科目内で表示順の重複を防止 |
| units | `(content_group_id, unit_number)` | 同一教材グループ内で単元番号の重複を防止 |
| student_subjects | `(student_id, subject_id)` | 同一生徒に同じ科目を二重登録しない |

#### 外部キーの ON DELETE 方針
| 親テーブル | 子テーブル | 方針 | 理由 |
|-----------|-----------|------|------|
| subjects | content_groups | `RESTRICT` | 教材グループが存在する科目は削除不可 |
| content_groups | units | `RESTRICT` | 単元が存在する教材グループは削除不可 |
| units | lesson_records | `RESTRICT` | 授業記録がある単元は削除不可 |
| students | lesson_records | `RESTRICT` | 授業記録がある生徒は物理削除不可（無効化 `is_active=false` で対応） |
| lesson_records | record_images | `CASCADE` | 授業記録を削除すると画像も連動削除（purge 操作時） |
| students | learning_goals | `CASCADE` | 生徒を完全削除する場合は目標も連動削除 |

#### パフォーマンスインデックス
| テーブル | カラム | 用途 |
|---------|--------|------|
| lesson_records | `(student_id, unit_id)` | 生徒×単元の進捗検索 |
| lesson_records | `(student_id, lesson_date DESC)` | 最近の授業記録表示 |
| units | `(content_group_id, unit_number)` | 教材グループ内の単元一覧取得 |

### 1.4 マイグレーション方針

> **現時点（プレローンチ）**: 本番データは存在しないため、`schema.sql` を新スキーマで書き直す。

## 2. 進捗ロジック

> **判定の基準**: 進捗ロジックは **教材グループの `category`**（vocabulary / academic）に基づいて分岐する。
> 同じ科目（例: 英語）内でも、教材グループごとに異なるフローが適用される。

### 2.1 完了判定基準

#### vocabulary 系（漢字・英単語・英熟語など）
```
対象:       content_groups.category = 'vocabulary' の教材グループに属する単元
通常フロー: ラーニング配布 → ステップ2テスト
完了条件:   ステップ2 の点数が満点の 80% 以上 → completion_type = 'passed'
再テスト:   80% 未満の場合、次回再テスト（再テスト上限なし）
ステップ1:  任意実施。実施時のみ記録（completion_type には影響しない）
```

#### academic 系（英文法・数学など）
```
対象:       content_groups.category = 'academic' の教材グループに属する単元
通常フロー: ラーニング → ステップ1 → (間違い再挑戦) → ステップ2
完了条件:   ステップ2 の点数が満点の 80% 以上 → completion_type = 'passed'
            ステップ1 で全問正解の場合、ステップ2 はスキップ可
              → ステップ1のレコードに completion_type = 'step1_perfect' を記録
再テスト:   80% 未満の場合、講師判断で再テストまたは次回に持ち越し
```

> **80%** は初期値。管理者が科目・生徒単位で変更可能にする（フェーズ2）。
> MVP（フェーズ1）では固定値として実装。

### 2.2 まとめテストのレコード分割ルール

複数単元をまとめてテストした場合（例: 第1〜4回を一括テスト→75/78）、以下のルールで記録する：

```
1. 対象の全単元に対して、それぞれ lesson_record を作成する
   - 例: 第1回, 第2回, 第3回, 第4回 の4レコード
   - すべて同じ step_type, score, max_score, lesson_date を設定
2. completion_type は通常ルール通り判定する（80%以上なら全単元 passed）
3. 答案画像（record_images）は対象範囲の最初の単元のレコードにのみ紐付ける
4. 全レコードの comment に固定フォーマットで範囲を記載する
   - フォーマット: `【まとめ】第1回〜第4回` （「【まとめ】」プレフィックス + 範囲）
   - システム側で自動付与し、講師が手動で書く必要はない
5. 同一テスト由来のレコードを束ねる group_id は持たない
   - comment の記載で十分に判別可能。発生頻度が低くDB設計を複雑にする価値がない
```

> **理由**: 1単元1レコードを維持することで、進捗計算・おすすめロジック・カレンダー表示が
> 特別な分岐なしで動作する。まとめテストは頻度が低いため、複雑な専用モデルは作らない。

#### 統計への影響ルール

| 指標 | 算出方法 | まとめテスト時の挙動 |
|------|---------|-------------------|
| 完了率 | 完了済み単元数 ÷ 全単元数 | 各単元が個別に完了扱い（正しく反映される） |
| 週ペース | 直近4週の完了単元数 ÷ 4 | 同日に4単元完了 = 4とカウント（実態通り） |
| 平均点 | 全step2レコードの score/max_score の平均 | **同一スコアが複数カウントされる**が、まとめテストは同じ実力で同じ範囲を解いた結果なので、複製による歪みは許容する。画面上に「※まとめテストを含む」の注記を表示 |

#### 答案画像の再表示仕様

まとめテストの画像は最初の単元のレコードにのみ紐付くが、他の単元からも参照できるようにする：

- **単元リスト（S-03）**: 画像が紐付いていない単元にはサムネイルを表示しない。ただし comment に「第1〜4回まとめテスト」の記載があるため、講師は最初の単元を見れば画像を確認できる
- **カレンダービュー（S-03-c）**: 日付タップで表示される記録一覧には、その日のすべてのレコードが表示される。画像が紐付いたレコードにはサムネイルが表示され、同日の他レコードからも視覚的に辿れる

### 2.3 次プリント提示ロジック

```
1. 生徒の受講科目に紐づく教材グループの単元一覧を取得
   （units → content_groups → subjects → student_subjects で絞り込み）
2. 各単元の lesson_records を確認し、教材グループごとに以下の優先順で提示:

   a. 再テスト対象（最優先）
      → step_type='step2' のレコードがあり completion_type = null（80%未満）
      → 「再テスト」として表示

   b. ラーニング済み・テスト未実施（vocabulary 系教材グループ）
      → step_type='learning' のレコードがあり step_type='step2' のレコードがない
      → 「ステップ2テスト」として表示

   c. ステップ1済み・ステップ2未実施（academic 系教材グループ）
      → step_type='step1' のレコードがあり step_type='step2' のレコードがない
      → 「ステップ2」として表示

   d. 未着手の単元（unit_number 昇順で最小）
      → レコードが一切存在しない単元
      → vocabulary 系:「ラーニング配布」、academic 系:「ラーニング開始」として表示

※ 「宿題ラーニング渡し済み」は lesson_records に step_type='learning' のレコードが
   存在するかどうかで判定する。専用フラグは持たない。
※ vocabulary / academic の判定は content_groups.category を参照する。
```

##### 手動フォールバックへの自動遷移条件

以下の**いずれか**に該当する場合、確認画面で手動選択UIを展開状態にし、警告メッセージを表示する。
自動遷移（画面切り替え）は行わない。確認画面内で修正できるようにする。

- AI解析の信頼度が **50%未満**
- 科目・教材・単元のいずれかがマッチしなかった（null）
- AI解析自体が失敗した（APIエラー）

> **50%** の閾値は、70%（従来の警告閾値）より厳しく設定。50〜70%の範囲では黄色警告を表示しつつAI値を初期表示し、50%未満では手動選択UIを展開する。

### 4.2 Gemini API 呼び出し仕様

#### 4.2.1 一括解析エンドポイント（写真優先フロー用）

```typescript
// POST /api/analyze
// リクエスト: multipart/form-data で画像ファイルを送信
// レスポンス:
{
  "subject_name": "英語",           // AIが読み取った科目名（null = 読み取れず）
  "content_group_name": "英文法",    // AIが読み取った教材名（null = 読み取れず）
  "unit_name": "【I】一般動詞",     // AIが読み取った単元名（null = 読み取れず）
  "step_type": "step2",             // AIが判定したステップ種別（null = 読み取れず）
  "score": 85,                      // 読み取った点数（null = 読み取れず）
  "max_score": 100,                 // 読み取った満点（null = 読み取れず）
  "confidence": 0.82,               // 全体の信頼度（0.0〜1.0）
  "raw_response": "..."             // Geminiの生レスポンス（デバッグ用）
}
```

#### 4.2.2 Gemini プロンプト仕様

```
この採点済みテスト用紙の画像を解析してください。

以下の情報をJSON形式で返してください：
1. subject_name: 科目名（「英語」「数学」「国語」のいずれか）
2. content_group_name: 教材名（プリントのタイトルやヘッダーから判読）
3. unit_name: 単元名（プリント上の単元表記をそのまま転記）
4. step_type: ステップ種別（"learning" / "step1" / "step2" のいずれか）
   - 「ラーニング」「Learning」→ "learning"
   - 「ステップ1」「Step1」「S1」→ "step1"
   - 「ステップ2」「Step2」「S2」→ "step2"
5. score: 採点された合計点数（数値）
6. max_score: 満点（数値）
7. confidence: 読み取りの信頼度（0.0〜1.0）

読み取れない項目はnullとしてください。
JSON以外のテキストは出力しないでください。
```

#### 4.2.3 フロントエンドでのマッチング処理

AIが返す `subject_name`, `content_group_name`, `unit_name` はプリント上の表記であり、
データベースの正確な値と一致しない可能性がある。フロントエンドで以下のマッチング処理を行う：

1. `subject_name` → `subjects` テーブルと**前方一致 or 部分一致**で照合
2. `content_group_name` → 該当科目の `content_groups` と部分一致で照合
3. `unit_name` → 該当教材グループの `units` と部分一致で照合
4. マッチしない場合 → 該当フィールドを未選択状態にし、手動選択を促す

> **補足**: AIの読み取り精度に依存しないよう、すべてのフィールドは確認画面で修正可能。
> AIが1項目でも読み取れれば、その分だけ講師の入力負荷が軽減される。

#### 4.2.4 部分一致マッチングの正規化ルール

照合前にAI出力値・DB値の両方に以下の正規化を適用する：

1. **全角英数→半角英数** に変換（例: `Ⅱ` → `II`、`１` → `1`）
2. **全角スペース→半角スペース**、連続スペースを1つに圧縮
3. **前後の空白をトリム**
4. **大文字→小文字** に統一（英字のみ）

照合アルゴリズム：
1. 正規化後の**完全一致**を最優先で検索
2. 一致なしの場合、正規化後の**部分一致**（DB値がAI出力値を含む、またはAI出力値がDB値を含む）
3. 部分一致が複数ヒットした場合、**最長一致**（一致文字数が最も多いもの）を採用
4. 同点の場合、`display_order`（教材グループ）または `unit_number`（単元）が小さい方を採用
5. いずれもマッチしない場合、null（未選択状態）

## 5. 個人情報の取り扱い

| 項目 | 決定事項 |
|------|---------|
| 保存するデータ | 氏名、学年、受講科目、学習記録（日時・点数）、答案画像、講師コメント |
| データ保存期間 | 退塾後 **12ヶ月** で管理者に削除通知。手動削除。 |
| 閲覧権限 | 管理者: 全生徒 / 講師: **全在籍生徒**（少人数のため講師制限は不要） |
| 答案画像 | Supabase Storage に保存。バケットは **private**（認証済みユーザーのみアクセス可） |
| 削除機能 | **2段階方式**: (1) 無効化（`is_active = false`、データ保持） → (2) 完全削除（`/purge`、記録・画像含む物理削除）。退塾後12ヶ月経過で管理者に完全削除を促す通知 |
| バックアップ | Supabase の自動バックアップに依存（Free プランは手動エクスポート） |
